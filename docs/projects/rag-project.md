# RAG 项目 — AI 面试助手

> 基于 Spring AI + pgvector 的 AI 面试助手平台，集成简历分析、模拟面试、RAG 知识库问答。

---

## 全局认知

### 业务架构

**用户痛点：**

- **简历质量难把控** — ToC 求职者不清楚自己简历的问题，ToB HR 筛选简历效率低
- **面试准备无针对性** — 通用面试题与实际简历不匹配，缺乏个性化练习
- **技术文档检索难** — 企业/项目内部知识分散，查询答案耗时

**解决方案：AI 面试助手**

- 简历管理（多格式上传、AI分析评分、改进建议、PDF报告导出）
- 模拟面试（AI 智能出题、实时问答交互、答案智能评估、面试报告生成）
- 知识库问答（文档上传与向量化、RAG检索增强、多轮对话问答、SSE流式响应）

### 技术架构

**核心数据流向**

| 场景 | 数据流向 | 关键技术 |
|:---|:---|:---|
| 简历上传分析 | 上传 → 文件解析(Tika) → MinIO存储 → Stream入队 → AI分析 → 结果入库 | 异步解耦、文件Hash去重 |
| 模拟面试 | 创建会话(Redis缓存) → AI出题 → 答题 → 分批评估 → 生成报告 | 分批处理防Token溢出 |
| RAG问答 | Query向量化 → pgvector相似度检索 → 上下文构建 → LLM生成 → SSE流式返回 | RAG检索增强 |

### 电梯演讲

**开场（10秒）：** "我独立开发了一个智能 AI 面试助手平台，帮助求职者和 HR 解决简历评估与面试准备的痛点。"

**技术亮点（90秒）：**

> "技术选型上采用 Spring Boot 3 + Java 21 + Spring AI 构建后端，前端用 React + TypeScript。四个核心亮点：
>
> 1. **异步解耦设计** — 简历分析、知识库向量化等耗时操作采用 Redis Stream 异步处理，避免 AI 调用阻塞用户请求，支持任务失败自动重试
> 2. **RAG 检索增强** — 基于 pgvector 实现向量检索，无需引入额外向量数据库；动态调整 TopK 和相似度阈值，自动扩展提高召回率
> 3. **流式交互体验** — 基于 SSE 实现打字机效果的流式响应，减少用户等待焦虑
> 4. **稳定性保障** — 面试评估采用分批处理策略（每批8题），规避大模型 Token 溢出风险；注解式限流组件防止 API 滥用"

## 模块拆解

### 简历管理模块

**核心流程：**

```
上传 → Tika解析 → MinIO存储 → Hash去重 → Redis Stream入队
→ Consumer消费 → AI分析 → 结果入库 → 用户查看报告
```

**关键技术实现：**

1. **文件解析（Apache Tika）** — 支持 PDF/DOCX/DOC/TXT/MD，自定义 `NoOpEmbeddedDocumentExtractor` 禁用嵌入资源解析避免提取PDF图片引用，`SortByPosition` 优化多栏布局
2. **异步处理（Redis Stream）** — 继承 `AbstractStreamConsumer`，专注业务逻辑，支持失败重试（最多3次）
3. **限流保护** — Redis + Lua 滑动窗口算法，`@RateLimit` 注解式使用，支持 GLOBAL + IP + USER 多维度
4. **结构化 Prompt** — System Prompt 定义角色和评分标准，`BeanOutputConverter` 强制 AI 返回 JSON 映射到 DTO
5. **文件 Hash 去重** — MD5 计算文件指纹，避免重复上传浪费 AI 调用成本

**面试话术：**

> "使用 Apache Tika 实现多格式文档解析时，我遇到了一个生产环境问题——PDF 中的图片被解析成了临时文件路径。解决方案是自定义 NoOpEmbeddedDocumentExtractor 禁用嵌入资源解析，只提取纯文本，同时设置 SortByPosition 优化多栏布局的文本顺序。"

### 模拟面试模块

**问题类型权重分配（后端面试）：**

| 类别 | 权重 | 说明 |
|:---|:---:|:---|
| 项目经历 | 20% | 深挖项目中的技术决策 |
| MySQL | 20% | 索引、事务、SQL优化 |
| Redis | 20% | 数据结构、缓存策略 |
| Java基础 | 10% | 集合、反射、泛型 |
| 集合 | 10% | HashMap、ConcurrentHashMap |
| 并发 | 10% | 线程池、锁、AQS |

**核心设计：**

1. **分类配比** — 按权重分配各类问题数量，贴近真实面试
2. **历史去重** — 传入历史问题列表，Prompt 中要求避免重复
3. **追问机制** — 每个主问题配置 1-2 条追问，构建线性追问流
4. **分批评估** — 每批 8 题独立调用 AI，避免单次 Token 超限，最后二次汇总生成整体评价
5. **断点续面** — 会话 Redis 缓存 24h，支持从数据库恢复

**面试话术（分批评估）：**

> "答案评估最大的挑战是 Token 溢出。假设一份简历有 20 道题，每道题的回答平均 500 字，加上简历本身和 Prompt，总 Token 可能超过 8k，超出模型限制。我的解决方案是 '分批评估 + 二次汇总'：将问题按每批 8 题分组，每批独立调用 AI 评估，合并各批次评估结果后，再调用一次 AI 对整体表现进行汇总（strengths、improvements、overallFeedback）。这样即使 50 道题也能稳定评估。"

### RAG 知识库模块

**知识库上传与向量化流程：**

```
文档上传 → Tika 文本提取 → TokenTextSplitter 分块(500 tokens/chunk)
→ 分批调用 Embedding API (1024维) → pgvector 向量存储
```

**RAG 问答流程：**

```
用户提问 → Query Rewrite(短查询扩展) → 向量相似度检索 → 有效性校验
→ Prompt 拼接(问题+检索文档) → SSE 流式返回
```

**检索优化策略：**

| 优化点 | 实现 |
|:---|:---|
| **动态检索参数** | 短查询 TopK=20/相似度=0.18（提高召回），长查询 TopK=8/相似度=0.28（提高精度） |
| **Query Rewrite** | "Redis" → "Redis 的数据结构和常用命令有哪些"，提高检索质量 |
| **有效性校验** | 短查询(2-20字符)要求检索结果必须包含查询关键词，避免"信息不足"的废话 |
| **探测窗口归一化** | 先缓冲前 120 字符，识别"无信息"模板则立即返回固定短句 |
| **多知识库联合检索** | `filter: kb_id IN ['1', '2']` 跨文档问答 |

**为什么用 pgvector 而不是专用向量数据库？**

> "架构精简，不引入新组件；PG 的向量功能已满足需求（HNSW 索引，千万级向量检索 ms 级）；事务一致性（向量和业务数据在一个数据库）"

### AI 配置管理

**多平台适配：**

| 维度 | 实现方式 |
|:---|:---|
| 配置存储 | PostgreSQL 持久化 + AES 加密 API Key |
| 工厂模式 | `AiChatClientFactory` 统一创建 ChatClient，封装不同平台差异 |

## 面试重点

### 高频问题

| 问题 | 回答要点 |
|:---|:---|
| "为什么 AI 分析要异步？" | AI 调用耗时 3-10 秒，同步会阻塞用户请求；异步提升用户体验，支持失败重试 |
| "如何保证分析不丢失？" | Redis Stream 持久化；消费者 ACK 确认；失败自动重试（最多3次） |
| "限流是怎么实现的？" | Redis + Lua 滑动窗口；原子操作避免竞态；多维度组合；注解式使用，零侵入 |
| "分批评估的具体策略？" | 默认每批8题；多轮调用 AI；合并后二次汇总生成整体评价 |
| "如何防止重复提问？" | 查询历史问题列表传入 Prompt；AI 生成时明确要求避免重复 |
| "为什么用 pgvector？" | 架构精简；PG HNSW 索引性能足够；事务一致性 |

### 亮点数据

- 向量维度：1024（text-embedding-v3）
- 分块大小：500 tokens/块，重叠 50 tokens
- 动态 TopK：短查询 20、中查询 12、长查询 8
- 相似度阈值：短查询 0.18、其他 0.28
- 分批评估：每批 8 题（可配置）
- 缓存 TTL：24 小时
- 支持文件格式：PDF/DOCX/DOC/TXT/MD
- 文件限制：最大 10MB

### 可能的追问

**Q: 如果 AI 服务挂了怎么办？**

> "Consumer 会捕获异常，标记任务为 FAILED，并保存错误信息。用户可以在前端看到失败状态，点击'重新分析'按钮手动触发重试。同时我们预留了降级策略的扩展点，可以对接本地 Ollama 模型作为备用。"

**Q: 如何防止用户刷接口？**

> "上传接口有 @RateLimit 注解，限制每个 IP 每分钟最多 5 次。使用 Redis + Lua 实现滑动窗口，相比固定窗口更平滑，避免窗口边界突发流量。"

**Q: RAG 检索失败怎么办？**

> "有 fallback 机制。如果带过滤条件的检索失败（可能是表达式语法问题），会降级到全量检索，然后在内存中过滤 kb_id，保证功能可用性。检索无结果时探测窗口识别并返回固定提示，避免大模型生成无意义的拒答。"
