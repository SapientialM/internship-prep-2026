## 1. 事务

### ACID 特性

| 特性 | 含义 | 实现方式 |
|:---|:---|:---|
| **A**tomicity（原子性） | 事务是最小执行单元，不可分割 | Undo Log（回滚日志） |
| **C**onsistency（一致性） | 事务前后数据完整性约束不变 | 由其他三个特性共同保证 |
| **I**solation（隔离性） | 并发事务间互不干扰 | MVCC + 锁 |
| **D**urability（持久性） | 事务提交后数据永久保存 | Redo Log（重做日志） |

### 并发事务带来的问题

| 问题 | 描述 |
|:---|:---|
| **脏读** | 读到其他事务未提交的数据 |
| **不可重复读** | 同一事务内两次读取同一条数据，结果不同（被其他事务 update 并提交） |
| **幻读** | 同一事务内两次查询，结果集行数不同（被其他事务 insert/delete 并提交） |

### 四种隔离级别

| 级别 | 脏读 | 不可重复读 | 幻读 | 实现 |
|:---|:---:|:---:|:---:|:---|
| READ UNCOMMITTED | ✅ | ✅ | ✅ | 无锁 |
| READ COMMITTED | ❌ | ✅ | ✅ | 每次读取生成 ReadView |
| REPEATABLE READ（默认） | ❌ | ❌ | ❌（大部分） | 事务开始时生成 ReadView |
| SERIALIZABLE | ❌ | ❌ | ❌ | 读写都加锁 |

**InnoDB 在 RR 级别下如何解决幻读？**

1. **MVCC（快照读）：** 普通 SELECT 通过 ReadView 读快照，看不到其他事务插入的数据
2. **Next-Key Lock（当前读）：** `SELECT ... FOR UPDATE` 等当前读，通过间隙锁（Gap Lock）+ 行锁（Record Lock）阻止其他事务插入

## 2. MVCC（多版本并发控制）

### 实现原理

MVCC 通过 ReadView + Undo Log 实现非锁定的一致性读。

**隐藏字段：**
- `DB_TRX_ID` — 最后修改该行的事务 ID（6 bytes）
- `DB_ROLL_PTR` — 回滚指针，指向 Undo Log（7 bytes）
- `DB_ROW_ID` — 隐藏主键（6 bytes，当没有显式主键时）

**ReadView 判断规则：**

```
当前事务 ID = cur_trx_id

遍历所有活跃事务，找到：
  - min_trx_id = 最小活跃事务 ID
  - max_trx_id = 最大活跃事务 ID + 1

每行数据 trx_id 的判断：
  - trx_id < min_trx_id  → 可见（在 ReadView 创建前已提交）
  - trx_id >= max_trx_id → 不可见（在 ReadView 创建后新开的事务）
  - 在两者之间 → 判断 trx_id 是否在活跃列表中
      - 在活跃列表中 → 不可见
      - 不在活跃列表中 → 可见
```

**RC vs RR 的 ReadView 区别：**
- RC：**每次 SELECT** 都生成新的 ReadView
- RR：**事务开始时**生成一次 ReadView，后续复用

## 3. 索引

### B+树索引

**为什么不用其他数据结构？**

| 数据结构 | 不适合的原因 |
|:---|:---|
| 哈希表 | 范围查询效率低；无法排序 |
| 二叉搜索树 | 可能退化为链表（O(n)）；深度太大导致大量磁盘 IO |
| B树 | 非叶子节点存储数据，导致每页存储的索引条目少，树更高 |
| 红黑树 | 二叉树变种，深度仍然大 |

**B+树的优势：**
1. 非叶子节点只存键值，每页可存更多索引，树更矮（减少磁盘 IO）
2. 叶子节点通过双向链表连接，支持高效范围查询
3. 数据只存储在叶子节点，查询效率稳定

### 聚簇索引 vs 非聚簇索引

| 类型 | 叶子节点存储 | 特点 |
|:---|:---|:---|
| **聚簇索引**（主键索引） | 完整行数据 | 一张表只有一个；InnoDB 自动按主键聚集 |
| **非聚簇索引**（二级索引） | 主键值 | 需要回表查主键索引获取完整行数据 |
| **覆盖索引** | — | 查询列都在索引中，不需要回表 |

**回表：** 二级索引找到主键，再用主键去聚簇索引找完整记录。避免回表的方式是使用覆盖索引。

### 索引失效的情况

1. **LIKE 前置模糊** — `LIKE '%xxx'` 无法使用索引
2. **WHERE 中使用函数/表达式** — `WHERE YEAR(create_time) = 2024`
3. **隐式类型转换** — `WHERE phone = 13800000000`（phone 是 varchar）
4. **OR 条件中有非索引列** — 每个 OR 分支都需要用索引才行
5. **NOT IN / NOT EXISTS / != / <>** — 大多数情况下不会走索引
6. **联合索引不满足最左前缀** — `INDEX(a, b, c)` 中查询 `b = ?` 不走索引
7. **范围查询右侧的列失效** — `WHERE a = 1 AND b > 10 AND c = 3`（c 不走索引）
8. **查询基数太小** — 优化器认为全表扫描更快（如性别字段）

## 4. 日志系统

### Redo Log（重做日志）

- **作用：** 保证事务的持久性，crash-safe
- **记录内容：** 物理日志，记录"在某页的某个位置做了什么修改"
- **写入方式：** 循环写（write pos 和 checkpoint 之间的空间可用）
- **刷盘策略（`innodb_flush_log_at_trx_commit`）：**
  - `0` — 每秒刷盘（可能丢1秒数据）
  - `1` — 每次提交刷盘（最安全）✅ 推荐
  - `2` — 每次提交写 page cache，每秒刷盘（OS 崩溃可能丢）

### Undo Log（回滚日志）

- **作用：** 事务回滚 + MVCC 读历史版本
- **记录内容：** 逻辑日志，记录数据修改前的值
- **生命周期：** 事务提交后不立即删除，直到没有其他事务需要它

### BinLog（归档日志）

- **作用：** 主从复制 + 数据恢复
- **记录内容：** 逻辑日志，记录 SQL 语句或行变更
- **写入方式：** 追加写，文件写满切换
- **刷盘策略（`sync_binlog`）：**
  - `0` — 由 OS 决定刷盘时机
  - `1` — 每次提交刷盘 ✅ 推荐
  - `N` — 每 N 次提交刷盘

### 两阶段提交（2PC）

Redo Log 和 BinLog 通过两阶段提交保证一致性：

```
1. Prepare Phase    → Redo Log 写 prepare 状态
2. Commit Phase     → BinLog 写入并刷盘
3.                  → Redo Log 写 commit 状态
```

崩溃恢复时：如果 Redo Log 是 prepare 状态且有对应的 BinLog → 提交；如果没有 BinLog → 回滚。

## 5. 锁

### 锁的类型

| 维度 | 类型 | 说明 |
|:---|:---|:---|
| 锁粒度 | 表锁 / 行锁 | InnoDB 支持行锁，MyISAM 仅支持表锁 |
| 锁模式 | 共享锁(S) / 排他锁(X) | S 锁兼容 S 锁，其他互斥 |
| 意向锁 | IS / IX | 表级锁，表示有行级 S/X 锁，用于判断表中是否有锁 |
| 间隙锁 | Gap Lock | 锁定索引间隙，防止插入 |
| 临键锁 | Next-Key Lock | Record Lock + Gap Lock，左开右闭区间 |
| 插入意向锁 | Insert Intention Lock | INSERT 时在间隙上设置的锁 |

### InnoDB 加锁规则

1. 只有通过索引检索数据时，才使用行锁；否则升级为表锁
2. RR 级别下，等值查询会退化为 Record Lock（唯一索引命中时）或 Next-Key Lock
3. 范围查询始终使用 Next-Key Lock

## 6. SQL 优化

### 执行计划 EXPLAIN

关键字段：

| 字段 | 说明 | 期望值 |
|:---|:---|:---|
| `type` | 访问类型 | system > const > eq_ref > ref > range > index > ALL |
| `key` | 实际使用的索引 | — |
| `rows` | 预估扫描行数 | 越小越好 |
| `Extra` | 额外信息 | 避免 Using temporary, Using filesort |

### 优化实战

1. **select * 能不用就不用** — 减少网络传输，增加覆盖索引机会
2. **合理分页** — 深分页用子查询/游标：`WHERE id > (SELECT id FROM t LIMIT 100000, 1) LIMIT 20`
3. **批量操作** — INSERT INTO ... VALUES (...), (...), (...)
4. **JOIN 代替子查询** — 小表驱动大表
5. **联合索引** — 区分度高的字段放前面
6. **避免使用 `%` 前置模糊查询** — 考虑用 Elasticsearch 等搜索引擎

### 慢查询分析流程

```
1. 开启慢查询日志（slow_query_log）
2. 设置 long_query_time 阈值
3. EXPLAIN 分析执行计划
4. 查看 type/rows/Extra 定位问题
5. 优化索引 / 改写 SQL / 数据归档
```
