# IM 即时通讯项目

> 基于 Netty 的即时通讯系统，支持单聊、群聊、消息推送、离线消息。

---

## 项目概述

### 核心功能

- **用户管理** — 注册、登录、好友管理
- **单聊** — 点对点实时消息
- **群聊** — 群组创建、加入退出、群消息
- **离线消息** — 用户离线时消息暂存，上线后推送
- **消息可靠性** — ACK 确认、消息去重、重发机制

### 技术栈

| 层级 | 技术 |
|:---|:---|
| 网络通信 | Netty（Reactor 多线程模型） |
| 协议 | TCP 长连接 + 自定义二进制协议 |
| 消息队列 | Redis Stream（离线消息存储） |
| 服务发现 | ZooKeeper |
| 数据存储 | MySQL + Redis |
| 架构 | Spring Boot |

### 长连接管理

**用户 → Channel 映射：**
```
Map<userId, Channel> userChannelMap  // 内存中维护
Redis: user:{userId}:server → serverIp:port  // 跨节点路由
```

**心跳检测：**

```java
// Netty IdleStateHandler 配置
pipeline.addLast(new IdleStateHandler(0, 0, 30, TimeUnit.SECONDS));
// 30 秒未收到任何数据，触发 USER_EVENT_IDLE → 关闭连接
```

**为什么不用 WebSocket？**

TCP 长连接 + 自定义二进制协议比 WebSocket 更高效：
- 二进制协议更紧凑（减少带宽）
- 更灵活的自定义编解码
- 更好的可控性（连接池、重连策略）

如果面向浏览器客户端，则必须使用 WebSocket。

### 消息可靠性保证

```
发送方 → 消息持久化(Redis) → 发送到接收方
        ↓ 未收到 ACK
        重试机制（最多 3 次）
```

- **ACK 确认：** 每条消息带 msgId，接收方收到后回 ACK，发送方超时未收到则重发
- **消息去重：** 客户端用 msgId 去重，防止重复推送
- **离线消息：** 接收方不在线时消息存 Redis List，上线后批量拉取

### 路由转发

**问题：** 用户 A 和用户 B 可能连接在不同 Netty 节点上。

**解决：**
1. 每个 Netty 节点在 ZooKeeper 注册临时节点：`/im/servers/{serverId}`
2. 用户登录时记录：`Redis: user:{userId}:server → serverId`
3. 发消息时先查接收方所在节点：
   - 本节点 → 直接发送
   - 其他节点 → 通过内部 HTTP/RPC 转发到目标节点

## 面试话术

### 高频问题

| 问题 | 回答要点 |
|:---|:---|
| "为什么用 Netty？" | NIO 多路复用、零拷贝、高性能吞吐；Reactor 线程模型；成熟的编解码器和处理器链 |
| "如何保证消息不丢？" | 消息持久化 + ACK 确认 + 超时重传 + 消息去重 |
| "离线消息怎么处理？" | Redis List 暂存离线消息，用户上线后批量拉取并一次性推送 |
| "服务怎么扩容？" | ZooKeeper 服务发现，新节点注册后自动加入集群；通过用户 ID 哈希分配连接 |
| "心跳为什么需要？" | 检测死连接（网络断开、客户端崩溃等极端情况），及时清理 Channel，释放资源 |

### 可能的追问

**Q: 如果有 10 万用户同时在线，如何优化？**

> "1) Netty Boss/Worker 线程池调整，Boss 线程 1-2 个，Worker 线程 = CPU 核数×2；2) 根据 userId hash 到不同 Netty 节点做水平扩展；3) 消息推送使用批量方式，减少 IO 次数；4) Redis 集群化，消息队列使用 Redis Cluster"

**Q: 如何保证消息有序？**

> "单聊场景：相同 sender→receiver 的消息加单调递增序号，接收方按序号排列。群聊场景：使用 Kafka 分区（同一群聊消息发到同一分区）保证分区内有序，或服务端加序列号。"
