# 03 - JVM

> JVM 是性能调优和问题排查的基础。内存模型、垃圾回收、类加载机制是高频考点。

---

## 1. JVM 内存结构（运行时数据区）

```
┌─────────────────────────────────────────┐
│              线程共享区域                 │
│  ┌─────────────┐  ┌───────────────────┐ │
│  │   堆 (Heap)  │  │ 方法区 (Metaspace) │ │
│  │  对象/数组   │  │ 类信息/常量/静态变量│ │
│  └─────────────┘  └───────────────────┘ │
├─────────────────────────────────────────┤
│              线程私有区域                 │
│  ┌────────┐ ┌────────┐ ┌──────────────┐ │
│  │程序计数器│ │虚拟机栈 │ │  本地方法栈   │ │
│  │ PC Reg  │ │VM Stack│ │ Native Stack│ │
│  └────────┘ └────────┘ └──────────────┘ │
└─────────────────────────────────────────┘
```

### 堆（Heap）

**分代模型（JDK 8）：**
- **新生代（Young）：** 默认占堆 1/3，包含 Eden + S0 + S1（比例 8:1:1）
- **老年代（Old）：** 默认占堆 2/3

**对象分配流程：**
```
new → Eden（不够触发 Minor GC）→ Survivor（来回拷贝，年龄+1）
→ 年龄达到阈值（默认15）→ 老年代 → Full GC 回收
```

**大对象直接进老年代：** 超过 `-XX:PretenureSizeThreshold` 的大对象直接在老年代分配，避免在 Eden/Survivor 之间来回复制。

### 虚拟机栈

每个线程创建时分配，包含多个**栈帧**（Stack Frame），每个方法调用对应一个栈帧：

| 组成部分 | 说明 |
|:---|:---|
| 局部变量表 | 存储方法参数和局部变量（基本类型+引用） |
| 操作数栈 | 字节码指令操作的临时数据 |
| 动态链接 | 指向运行时常量池中该方法的引用 |
| 返回地址 | 方法出口 |

**StackOverflowError vs OutOfMemoryError：**
- 线程请求的栈深度 > JVM 允许的最大深度 → `StackOverflowError`
- 创建过多栈帧（如递归过深）或线程数过多，内存不够创建新栈 → `OutOfMemoryError`

### 方法区（Metaspace）

JDK 8 用 Metaspace（本地内存）替代永久代（PermGen）：

| 特性 | PermGen (JDK 7) | Metaspace (JDK 8+) |
|:---|:---|:---|
| 存储位置 | JVM 堆内 | 本地内存（Native Memory） |
| 大小限制 | `-XX:MaxPermSize` | `-XX:MaxMetaspaceSize`（默认无限制） |
| 溢出异常 | `java.lang.OutOfMemoryError: PermGen space` | `java.lang.OutOfMemoryError: Metaspace` |

存放内容：类信息、运行时常量池、静态变量、JIT 编译后的代码缓存。

## 2. 对象的创建与访问

### 创建对象的过程

```
1. 类加载检查（检查是否已被加载、解析、初始化）
2. 分配内存（指针碰撞 / 空闲列表，取决于 GC 是否有压缩）
3. 初始化零值（保证不赋初值就能使用）
4. 设置对象头（Mark Word + 类型指针）
5. 执行 <init> 方法（构造器 + 初始化块）
```

### 对象一定分配在堆里吗？

不一定。JIT 编译时，如果对象不会逃逸出方法，可以通过**逃逸分析**在**栈上分配**，减少 GC 压力。另外虚拟对象（标量替换）也可以直接拆分为基本类型在栈上分配。

### 对象的内存布局

```
┌───────────────┐
│   对象头       │ Mark Word (8B) + Klass Pointer (4B/8B)
├───────────────┤
│   实例数据     │ 字段（含父类字段）
├───────────────┤
│   对齐填充     │ 保证 8 字节对齐
└───────────────┘
```

**Mark Word：** 存储对象自身的运行时数据（HashCode、GC分代年龄、锁状态标志、线程持有的锁、偏向线程ID等），在 32/64 位 JVM 中分别为 32/64 bit。

## 3. 垃圾回收

### 判断对象是否可回收

**根可达性分析（GC Roots）：**

从 GC Roots 对象出发，沿着引用链向下搜索，无法到达的对象判定为可回收。

**GC Roots 包括：**
1. 虚拟机栈中引用的对象（局部变量）
2. 方法区中静态变量引用的对象
3. 方法区中常量引用的对象
4. 本地方法栈中 JNI 引用的对象
5. 所有被同步锁（synchronized）持有的对象
6. Java 虚拟机内部的引用（基本数据类型的 Class 对象、常驻异常、类加载器等）

### 垃圾回收算法

| 算法 | 原理 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **标记-清除** | 标记存活对象，清除未标记的 | 简单 | 产生内存碎片 |
| **标记-复制** | 将存活对象复制到另一块区域 | 无碎片，效率高 | 浪费一半空间 |
| **标记-整理** | 标记后将存活对象移向一端 | 无碎片 | 移动对象开销大，STW |
| **分代收集** | 新生代用复制，老年代用标记-整理 | 兼顾效率和空间 | 实现复杂 |

### 垃圾回收器

| 回收器 | 收集区域 | 算法 | 特点 | 适用场景 |
|:---|:---|:---|:---|:---|
| Serial | 新生代 | 复制 | 单线程，STW 长 | 客户端应用 |
| ParNew | 新生代 | 复制 | 多线程并行 | CMS 搭档 |
| Parallel Scavenge | 新生代 | 复制 | 吞吐量优先 | 后台计算 |
| Serial Old | 老年代 | 标记-整理 | 单线程 | 客户端 |
| CMS | 老年代 | 标记-清除 | 低停顿目标 | 互联网应用 |
| G1 | 全部 | 标记-整理+复制 | 可控停顿 | 大堆（>4G） |
| ZGC | 全部 | 标记-复制（染色指针） | 亚毫秒停顿 | 低延迟，大堆 |
| Shenandoah | 全部 | 标记-复制 | 低停顿 | 大堆 |

**CMS 的缺点：**
1. 并发阶段仍占用 CPU 资源
2. 标记-清除算法会产生内存碎片
3. 无法处理浮动垃圾（并发清理阶段新产生的垃圾）
4. JDK 14 已废弃，JDK 15 移除

**G1 的特点：**
- 将堆划分为大小相等的 Region（默认 2048 个）
- 每次回收价值最高的 Region（Garbage First）
- 通过 `-XX:MaxGCPauseMillis` 设定停顿目标
- 使用 Remembered Set 解决跨 Region 引用问题

### 三色标记法

CMS 和 G1 使用三色标记法进行并发标记：

- **白色：** 未被访问（标记结束后白色对象被回收）
- **灰色：** 已访问但引用尚未扫描
- **黑色：** 已访问且引用已扫描

**漏标问题（黑色对象引用了白色对象）：**

解决方式：
- **增量更新（CMS）：** 黑色对象新增白色引用时，将黑色重新标记为灰色
- **原始快照 SATB（G1）：** 灰色对象删除白色引用时，记录该白色对象

## 4. 类加载机制

### 类的生命周期

```
加载 → 验证 → 准备 → 解析 → 初始化 → 使用 → 卸载
```

### 双亲委派模型

```
Bootstrap ClassLoader (加载核心类 rt.jar)
         ↑
Extension ClassLoader (加载扩展类 jre/lib/ext)
         ↑
Application ClassLoader (加载 classpath 下的类)
         ↑
自定义 ClassLoader
```

**工作流程：**
1. 收到加载请求时，先委托父类加载器
2. 父类无法加载时，子类才尝试加载

**为什么需要双亲委派？**
1. 安全性 — 防止核心类（如 `java.lang.Object`）被篡改
2. 避免重复加载 — 同一个类只被加载一次

**如何打破双亲委派？**

重写 `findClass()` 方法而不是 `loadClass()` 方法。例如 Tomcat 为了支持多个 Web 应用隔离，使用自定义 ClassLoader 打破双亲委派。

## 5. JVM 调优

### 常用调优参数

| 参数 | 说明 | 示例 |
|:---|:---|:---|
| `-Xms` | 初始堆大小 | `-Xms2g` |
| `-Xmx` | 最大堆大小 | `-Xmx4g` |
| `-Xmn` | 新生代大小 | `-Xmn1g` |
| `-XX:+UseG1GC` | 使用 G1 垃圾回收器 | — |
| `-XX:MaxGCPauseMillis` | 期望最大 GC 停顿 | `-XX:MaxGCPauseMillis=200` |
| `-XX:+PrintGCDetails` | 打印 GC 详细日志 | — |
| `-XX:+HeapDumpOnOutOfMemoryError` | OOM 时 Dump 堆 | — |

### 调优思路

1. 先看 GC 日志，确定是 Minor GC 频繁还是 Full GC 频繁
2. Minor GC 频繁 → 调大新生代
3. Full GC 频繁 → 分析老年代增长原因（大对象、内存泄漏）
4. STW 过长 → 考虑换 GC 回收器（G1/ZGC）

### 常见 OOM 场景

| 异常 | 原因 |
|:---|:---|
| `java.lang.OutOfMemoryError: Java heap space` | 堆内存不足 |
| `java.lang.OutOfMemoryError: Metaspace` | 加载太多类 |
| `java.lang.OutOfMemoryError: unable to create new native thread` | 线程数过多 |
| `java.lang.StackOverflowError` | 栈溢出（递归过深） |
