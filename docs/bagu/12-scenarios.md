# 12 - 场景设计题

> 场景题考察系统设计能力和技术广度的综合运用。没有标准答案，关键是展现分析思路和权衡能力。

---

## 1. 如何与外部合作方交互防止被拖垮？

**核心策略：隔离 + 熔断 + 降级 + 限流。**

```
┌─────────────────────────────────────────────────┐
│               我方服务                            │
│  ┌──────────┐  ┌──────────┐  ┌──────────────┐  │
│  │  限流器   │  │  熔断器   │  │  降级逻辑     │  │
│  └─────┬────┘  └─────┬────┘  └──────┬───────┘  │
│        │              │              │          │
│        ▼              ▼              ▼          │
│  ┌──────────────────────────────────────────┐   │
│  │              线程池隔离                    │   │
│  └──────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
                         │
                         ▼
              ┌──────────────────┐
              │   外部合作方服务   │
              └──────────────────┘
```

**具体措施：**

1. **线程池隔离** — 为外部调用分配独立的线程池，不共享主业务线程池
2. **超时控制** — 设置合理的 connectTimeout + readTimeout（如连接 2s，读取 5s）
3. **熔断保护** — 用 Hystrix/Sentinel，失败率超过阈值时熔断，快速失败
4. **降级策略** — 外部服务不可用时返回默认值/缓存数据/空列表
5. **限流** — 对调用外部服务的接口做 QPS 限制
6. **异步解耦** — 非关键路径的调用通过 MQ 异步处理

## 2. 如何实现 A 网站登录后 B 网站自动登录？（单点登录 SSO）

**方案一：JWT + 同域名 Cookie**

```
1. A 登录 → 认证中心验证 → 返回 JWT → 种在顶级域名的 Cookie (.example.com)
2. B 请求携带该 Cookie → B 后端解密 JWT 验证 → 获取用户信息自动登录
```

**方案二：CAS 协议（经典 SSO）**

```
1. 用户访问 A → 重定向到认证中心（/login?service=A）
2. 认证中心验证用户 → 签发 TGT(全局会话) + ST(服务票据)
3. 重定向回 A，携带 ST
4. A 后端拿 ST 去认证中心验证 → 验证通过则登录
5. 用户访问 B → 重定向到认证中心（已有 TGT）→ 直接签发 ST 跳回 B
```

**方案三：OAuth 2.0 / OIDC**

主要用在第三方登录（微信/Google 登录），适合跨公司授权场景。

## 3. 如何预估接口上线后的 QPS？

**方法一：根据 DAU 估算**

```
QPS = DAU × 人均请求次数 ÷ (活跃小时数 × 3600) × 峰值系数

例：
  DAU = 10万
  人均请求 = 20次
  活跃时段 = 8小时
  峰值系数 = 3（晚间高峰是均值的 3 倍）

  QPS ≈ 100000 × 20 ÷ (8 × 3600) × 3 ≈ 208 QPS
```

**方法二：根据已有业务量推断**

```
新接口 QPS = 关联接口 QPS × 调用比例

例：下单接口的 QPS 约 = 商品详情页 QPS × 转化率
```

**方法三：压力测试（最准确）**

上线前在预发布环境（或低峰期生产环境）做压测，逐步增加并发数，找到 QPS 拐点。

## 4. 订单到期关闭如何实现？

| 方案 | 原理 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **延迟队列** | RocketMQ 延时消息 / RabbitMQ 死信队列 | 可靠，生产级 | 依赖 MQ |
| **Redis 过期回调** | `keyspace notifications` 监听过期事件 | 简单 | 不可靠（Redis key 删除时才通知，可能漏） |
| **定时任务轮询** | 扫描待支付订单，到期更新状态 | 实现简单 | 数据量大时性能差，需要分批 |
| **Redisson 延迟队列** | `RDelayedQueue` 基于 Redis 实现 | Java 生态友好 | 需要 Redisson |
| **时间轮** | Netty HashedWheelTimer / Kafka 时间轮 | 高效 | 需要自己实现可靠性 |

**推荐方案：** RocketMQ 延时消息（18 个延时级别，如 level=16 对应 30min）+ 定时任务兜底扫表。

## 5. 亿级数据取出最大前 N 个

### 小顶堆（单机）

```
1. 维护一个大小为 N 的小顶堆
2. 遍历所有数据
3. 如果当前元素 > 堆顶，替换堆顶并调整堆
4. 遍历完成后，堆中就是最大的 N 个

时间复杂度：O(M × logN)，M 为数据总量
空间复杂度：O(N)
```

### 分治 + 归并（大数据量）

```
1. 分片：将数据分割，每片找到 Top N
2. 归并：将所有分片的 Top N 汇总，再取全局 Top N
```

### MapReduce（超大）

Map 阶段每个节点独立找 Top N，Reduce 阶段汇总取全局 Top N。

## 6. 如何设计一个秒杀系统？

```
用户请求 → CDN（静态资源）→ 限流 → 秒杀校验 → 异步下单
              │                  │         │          │
              ▼                  ▼         ▼          ▼
      减少服务器压力        防刷/防超     扣减库存   MQ异步处理
```

**核心设计：**

1. **前端限流** — 按钮置灰 + 验证码 + 答题
2. **网关层限流** — Nginx 令牌桶/Sentinel 限流
3. **Redis 库存预扣减** — Lua 脚本原子扣库存（比 DB 快 100 倍）
4. **异步下单** — 扣库存成功后，通过 MQ 异步创建订单
5. **防超卖** — Redis Lua 原子操作 + DB 乐观锁（version 字段）兜底
6. **流量削峰** — MQ 排队下单，平滑后端处理压力
7. **降级兜底** — 秒杀结束后显示"已抢完"

### 秒杀场景下如何增加库存？

不建议在秒杀进行中增加库存（破坏公平性）。如果必须增加：
1. 暂停秒杀活动
2. 运营后台同步增加 Redis 和 DB 中的库存数量
3. 重新开放秒杀
