# 14 - 分库分表

> 当单表数据量达到千万级，MySQL 的性能会明显下降。分库分表是大型系统的标配方案。

---

## 1. 什么时候需要分库分表？

| 问题 | 表现 | 解决方案 |
|:---|:---|:---|
| 单表数据量大（> 2000w） | 慢查询、索引大、重建慢 | 水平分表 |
| 单库连接数高/并发高 | 数据库 CPU 高、连接池耗尽 | 垂直分库 |
| 写吞吐量瓶颈 | 写入 TPS 达到上限 | 水平分库分表 |
| 存储容量不足 | 磁盘接近极限 | 分库（分散到多台） |

**优先考虑优化而非分库：** 索引优化、SQL 优化、读写分离、缓存、数据归档。这些手段都用完还不够才考虑分库分表。

---

## 2. 垂直分库 vs 水平分库

### 垂直分库

按业务模块拆分不同库：用户库、订单库、商品库。

**优点：** 业务边界清晰，独立扩容。
**缺点：** 跨库 JOIN 无法使用，分布式事务。

### 水平分库分表

将同一张表的数据按一定规则拆分到多个库/表中。

**优点：** 单表数据量下降，写入吞吐量提升。
**缺点：** 查询复杂，需要路由层。

---

## 3. 分片策略

### 分片键选择原则

1. **高区分度** — 数据能均匀分散到各分片（避免热点）
2. **查询命中** — 大部分查询能带上分片键
3. **避免跨分片** — 减少分布式 JOIN / 跨分片聚合

### 常用分片算法

| 算法 | 示例 | 优点 | 缺点 |
|:---|:---|:---|:---|
| **取模** | `hash(key) % N` | 数据分布均匀 | 扩容时需要全量迁移 |
| **范围分片** | `order_id ∈ [1, 1000w] → 分片1` | 扩容方便 | 热点数据集中（最新数据在同一分片） |
| **一致性哈希** | 虚拟节点映射到哈希环 | 扩容迁移量小 | 实现复杂 |
| **复合分片** | 先范围再取模 | 灵活 | 复杂 |

## 4. 分库分表后的挑战

### 分布式主键

雪花算法生成全局唯一 ID（趋势递增，保证 MyISAM/InnoDB 写入性能）。

### 跨分片查询

| 场景 | 解决方案 |
|:---|:---|
| 按非分片键查询 | ES 索引（先查 ES 拿到分片键，再查 DB）+ MySQL 异构索引表 |
| 跨分片分页 | 全局查询（每个分片取前 N 条，汇总再排序）→ 性能优化：使用 ES |
| 跨分片 JOIN | 应用层 JOIN / 字段冗余 / 全局表 |
| 跨分片聚合（count/sum/avg） | 各分片执行 → 应用层汇总（或使用数据仓库如 ClickHouse） |

### 分布式事务

使用最终一致性方案（本地消息表 / RocketMQ 事务消息 / Seata Saga 模式）。

### 扩容与数据迁移

**方案一：翻倍扩容（推荐）**

```
分片 N → 分片 2N
迁移：hash(key) % N 中一半的数据迁移到新分片
迁移量：50%

例：2 分片 → 4 分片
  原来 hash%2=0 的数据，一半(hash%4=0)留在原分片，一半(hash%4=2)迁移
```

**方案二：一致性哈希**

迁移量：约 1/N。

**数据迁移流程：**

1. 双写（新旧表都写）+ 异步数据迁移
2. 数据校验（行数、关键数据抽样对比）
3. 灰度切换读流量
4. 停止写旧表

## 6. 什么时候不该分库分表

1. 数据量不超过千万
2. 单表 QPS 远低于数据库承载上限
3. 业务逻辑非常复杂（关联查询极多）
4. 团队规模小（分库分表引入的运维成本远超性能收益）

先尝试：**优化索引 → 读写分离 → 缓存 → 冷热数据分离 → 数据归档 → 最后才分库分表**。
