## 1. HTTP & HTTPS

### HTTP 各版本演进

| 版本 | 特点 |
|:---|:---|
| HTTP/1.0 | 短连接，每次请求建立 TCP 连接 |
| HTTP/1.1 | 持久连接（Keep-Alive）、管道化、Host 头、断点续传 |
| HTTP/2.0 | 多路复用、头部压缩（HPACK）、服务器推送、二进制分帧 |
| HTTP/3.0 | 基于 QUIC（UDP）、0-RTT 握手、无队头阻塞 |

### HTTPS 握手流程

```
1. 客户端 → 服务器: ClientHello（支持的 TLS 版本 + 加密套件 + 随机数1）
2. 服务器 → 客户端: ServerHello（选定的加密套件 + 随机数2）+ 证书
3. 客户端验证证书（CA 签名 + 域名）
4. 客户端生成 Pre-Master Secret，用服务器公钥加密发送
5. 双方用 随机数1 + 随机数2 + Pre-Master Secret 生成会话密钥
6. 后续通信使用对称密钥加密
```

### HTTP 状态码

| 范围 | 含义 | 常见状态码 |
|:---|:---|:---|
| 1xx | 信息 | 101 Switching Protocols（WebSocket 升级） |
| 2xx | 成功 | 200 OK, 201 Created, 204 No Content |
| 3xx | 重定向 | 301 永久重定向, 302 临时重定向, 304 Not Modified |
| 4xx | 客户端错误 | 400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests |
| 5xx | 服务器错误 | 500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable, 504 Gateway Timeout |

### GET vs POST

| 维度 | GET | POST |
|:---|:---|:---|
| 语义 | 获取资源 | 创建/修改资源 |
| 参数位置 | URL 查询字符串 | 请求体 |
| 缓存 | 可缓存 | 不可缓存（默认） |
| 幂等 | 是 | 否 |
| 安全性 | 参数暴露在 URL | 参数在请求体（相对安全） |
| 长度限制 | 受 URL 长度限制（浏览器/服务器限制） | 无明确限制 |

## 2. TCP

### TCP 三次握手

```
客户端                        服务器
  │ ------ SYN (seq=x) --------→ │  第一次：客户端发 SYN
  │ ←-- SYN+ACK (seq=y, ack=x+1) │  第二次：服务器回 SYN+ACK
  │ ------ ACK (ack=y+1) ------→ │  第三次：客户端确认
  │        连接建立               │
```

**为什么是三次而不是两次？**

防止已失效的连接请求到达服务器后，服务器误以为新连接。如果只有两次握手，服务器收到迟到的 SYN 会直接建立连接，浪费资源。

### TCP 四次挥手

```
客户端                        服务器
  │ ------ FIN (seq=u) --------→ │  第一次：客户端发 FIN
  │ ←-- ACK (ack=u+1) ----------│  第二次：服务器确认
  │ ←-- FIN (seq=v, ack=u+1) ---│  第三次：服务器发 FIN
  │ ------ ACK (ack=v+1) ------→ │  第四次：客户端确认
  │  等待 2MSL 后关闭               │
```

**为什么是四次？**

TCP 是全双工的，每个方向需要单独关闭。服务器收到 FIN 后可能还有数据要发送，所以 ACK 和 FIN 分开发送。

**为什么需要 TIME_WAIT（2MSL）？**

1. 确保最后的 ACK 能被服务器收到（如果 ACK 丢失，服务器会重发 FIN）
2. 让旧连接的报文在网络中消失，避免影响新连接

### TCP 可靠性保证

| 机制 | 说明 |
|:---|:---|
| **序列号 + 确认应答** | 每个字节编号，接收方确认已收到的字节 |
| **超时重传** | 超过 RTO（重传超时时间）未收到 ACK，重发数据 |
| **流量控制** | 滑动窗口，接收方告知剩余缓冲区大小 |
| **拥塞控制** | 慢启动 → 拥塞避免 → 快速重传 → 快速恢复 |
| **校验和** | 检测数据在传输中的错误 |

### TCP 粘包和拆包

- **粘包：** 多个小包被合并为一个（Nagle 算法导致）
- **拆包：** 大包被拆分为多个 TCP 段（MSS 限制）

**解决方式：**
1. 固定长度消息（适用于固定长度协议）
2. 分隔符（如 `\r\n`，HTTP 使用）
3. 消息头 + 消息体（消息头包含 body 长度）

## 3. 键入网址到网页显示，期间发生了什么？

```
1. URL 解析 — 提取协议、域名、路径
2. DNS 解析 — 域名 → IP 地址（浏览器缓存 → hosts → DNS 服务器 递归/迭代查询）
3. TCP 连接 — 三次握手建立连接（HTTPS 再进行 TLS 握手）
4. 发送 HTTP 请求 — 构建请求报文并发送
5. 服务器处理 — 接收请求并返回响应
6. 浏览器解析渲染 — HTML → DOM Tree, CSS → CSSOM → Render Tree → Layout → Paint
7. TCP 断开 — 四次挥手（Connection: keep-alive 则保持连接）
```

## 4. CDN

### 工作原理

```
用户 → 就近 CDN 边缘节点 → 缓存命中返回 → 未命中回源（源站）→ 缓存并返回
```

### 核心机制

1. **DNS 智能解析** — 根据用户 IP 返回最近的 CDN 节点 IP
2. **内容缓存** — 静态资源缓存在边缘节点
3. **缓存更新** — 通过 URL 版本号 / hash / CDN 刷新接口

### CDN 对业务的作用

- 加速静态资源（JS/CSS/图片）
- 减少源站带宽压力
- 防御部分 DDoS 攻击
